# SPEC-PREVIEW-004 — 수용 기준

> 검증 계획 출처: `docs/design/html-preview.md` 6장. 보안 시나리오(3·4·5)와 회귀 시나리오(7)는 **반드시 통과**해야 하는 must-pass 기준이다.

## 사전 준비: 테스트용 샘플 폴더

다음을 포함하는 샘플 작업 폴더 하나를 구성한다.

- 정상 HTML 파일 1개 + 그것이 참조하는 같은 폴더의 CSS·이미지·스크립트
- 하위 폴더의 또 다른 HTML 파일 (상위 폴더 자산을 참조)
- 공격 시뮬레이션용 HTML (앱 권한 접근 시도)
- 경로 탈출 시도용 HTML (`../../` 등으로 작업 폴더 밖 접근 시도)
- 범위 밖 파일 읽기 시도용 HTML
- 5MB를 초과하는 대용량 HTML 파일 1개

---

## 기능 시나리오

### 시나리오 1: 사이드바에 HTML 파일이 표시되고 클릭 시 브라우저처럼 동작 (REQ-PREVIEW004-001, 004-002, 004-004)

- **Given** 샘플 폴더가 사이드바에 열려 있고
- **When** 사용자가 사이드바를 보고 정상 HTML 파일을 클릭하면
- **Then** 사이드바 트리에 `.html` 파일이 `.md` 파일과 함께 표시되고
- **And** 프리뷰 칸이 HTML 보기 모드로 전환되어 샌드박스 iframe 안에서 HTML이 펼쳐지고
- **And** HTML의 디자인(CSS)·이미지·움직이는 기능(스크립트)이 실제 웹브라우저처럼 동작하고
- **And** 편집기 패널에는 "이 형식은 편집할 수 없습니다" 보기 전용 플레이스홀더가 표시된다.

### 시나리오 2: 마크다운 ↔ HTML 전환 및 앱 셸 격리 (REQ-PREVIEW004-001, 004-002)

- **Given** HTML 파일이 HTML 보기 모드로 표시되어 있고
- **When** 사용자가 사이드바에서 `.md` 파일을 선택했다가 다시 `.html` 파일을 선택하면
- **Then** 프리뷰 칸이 마크다운 렌더링 모드와 HTML 보기 모드 사이에서 올바르게 전환되고
- **And** 편집기는 마크다운일 때 편집 가능, HTML일 때 보기 전용 플레이스홀더로 올바르게 바뀌고
- **And** HTML의 스타일·레이아웃이 iframe 경계를 넘어 앱 셸(헤더·사이드바·푸터)을 변형시키지 않는다.

### 시나리오 3 (기능): 하위 폴더 HTML이 상위 폴더 자산 로드 (REQ-PREVIEW004-003)

- **Given** 폴더 A가 열려 있어 Rust 백엔드가 `allow_directory(A, true)`로 `A/**`를 런타임 등록했고, A의 하위 폴더에 상위 폴더 자산을 참조하는 HTML 파일이 있고
- **When** 사용자가 그 하위 폴더 HTML을 클릭하면
- **Then** 상위 폴더의 CSS·이미지가 `A/**` glob 범위 안에 있으므로 정상적으로 로드되어 표시된다.

## 보안 시나리오 (must-pass — `docs/design/html-preview.md` 6장)

### 보안 시나리오 3: 앱 권한 탈취 시도 차단 (REQ-PREVIEW004-005)

- **Given** "앱의 파일 읽기·쓰기·삭제 권한을 사용하려 시도하는" 공격 시뮬레이션 HTML 파일이 샘플 폴더에 있고
- **When** 사용자가 그 파일을 HTML 보기 모드로 연 뒤 그 안의 스크립트가 실행되면
- **Then** iframe 격리로 인해 앱 권한 호출이 차단되어 어떤 파일도 읽히거나 수정·삭제되지 않고
- **And** 앱 본체는 정상 상태를 유지한다.

### 보안 시나리오 4: 경로 탈출 시도 차단 (REQ-PREVIEW004-005)

- **Given** 폴더 A가 열려 있어 `A/**`가 런타임 등록된 상태에서, A 안의 HTML 파일이 `../../B/secret` 처럼 A 밖 형제 폴더 B의 파일을 참조하고
- **When** 사용자가 그 HTML 파일을 열면
- **Then** `../../B/secret`이 해소된 절대 경로는 `A/**` glob에 매칭되지 않으므로 asset 프로토콜 scope에 의해 요청이 차단되어 해당 자산이 로드되지 않는다.
- **And** A 폴더 경로가 정규화(심링크 해소)되어 등록되었으므로 심링크를 통한 우회도 차단된다.

### 보안 시나리오 5: 범위 밖 파일 읽기 시도 차단 (REQ-PREVIEW004-005)

- **Given** 폴더 A가 열려 있어 `A/**`만 런타임 등록된 상태에서, A 안의 HTML 파일이 `/etc/passwd`(또는 사용자 홈의 비밀 파일) 같은 절대 경로를 참조하고
- **When** 사용자가 그 HTML 파일을 열고 스크립트가 그 절대 경로 접근을 시도하면
- **Then** 그 경로는 등록된 어떤 허용 패턴(`A/**`)에도 매칭되지 않으므로 차단되어 범위 밖 파일 내용이 노출되지 않는다.

### 보안 시나리오 5-A: 폴더 미오픈 시 빈 scope (REQ-PREVIEW004-003)

- **Given** 앱이 막 재시작되어 아직 어떤 폴더도 열리지 않은 상태에서
- **When** 임의의 로컬 파일을 asset 프로토콜로 요청하면
- **Then** 정적 `scope`가 빈 배열이고 런타임 등록도 없으므로 asset 프로토콜 scope가 비어 있어 어떤 로컬 파일도 전달되지 않는다.

### 회귀 시나리오 7: 기존 마크다운 기능 회귀 없음 (REQ-PREVIEW004-003 부수)

- **Given** `tauri.conf.json`에 `assetProtocol: { "enable": true, "scope": [] }` 블록이 추가되고 폴더 열기 커맨드에 `allow_directory` 등록이 추가된 상태에서
- **When** 사용자가 기존 마크다운 파일(코드 하이라이트·KaTeX 수식·Mermaid 다이어그램 포함)을 열면
- **Then** 마크다운 렌더링(SPEC-PREVIEW-001)·스크롤 싱크(SPEC-PREVIEW-002)·수식(SPEC-PREVIEW-003)이 모두 변경 전과 동일하게 정상 동작하고
- **And** 기존 이미지 표시 및 export 기능이 회귀 없이 동작한다.

## 성능 시나리오

### 성능 시나리오 6: 대용량 HTML 파일이 앱을 멈추지 않음 (REQ-PREVIEW004-005)

- **Given** 5MB를 초과하는 대용량 HTML 파일이 샘플 폴더에 있고
- **When** 사용자가 그 파일을 클릭하면
- **Then** 시스템은 iframe 렌더링을 수행하지 않고 "미리보기 불가" 안내를 표시하고
- **And** 앱이 멈추거나 응답 불가 상태가 되지 않는다.

### 성능 시나리오 8: 빠른 반복 전환 시 메모리 누수 없음

- **Given** 마크다운 파일과 HTML 파일이 모두 사이드바에 있고
- **When** 사용자가 마크다운 ↔ HTML 보기를 빠르게 여러 번 반복 전환하면
- **Then** iframe이 매 전환마다 정리되어 메모리가 누적되지 않고
- **And** 전환이 계속 정상적으로 동작한다.

---

## 엣지 케이스

- 자산 파일이 없는 HTML(깨진 이미지 링크): 앱이 멈추지 않고 HTML 자체는 계속 표시된다.
- 0바이트 / 빈 HTML 파일: 빈 iframe이 표시되고 오류로 처리되지 않는다.
- 확장자가 `.HTML`처럼 대문자인 파일: `.html`과 동일하게 HTML 보기 모드로 처리된다(`toLowerCase` 비교).
- 파일 읽기 실패 / iframe 로드 실패: 앱 중단 없이 오류 안내가 표시된다.
