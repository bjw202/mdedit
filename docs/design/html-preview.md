# HTML 파일 보기 기능 추가 — 설계문서

## 1. 한눈에 보기

지금 이 앱(`mdedit`)은 **마크다운 파일(.md) 전용 편집기**입니다. 여기에 **HTML 파일(.html)을 웹브라우저처럼 보여주는 기능**을 추가하려고 합니다.

확정된 범위 (사용자와 합의):

| 항목 | 결정 | 의미 |
| --- | --- | --- |
| 편집 여부 | **보기 전용** | HTML 파일은 "보기"만 가능. 고치는 건 마크다운만. |
| 스크립트 동작 | **실행함** | HTML 안의 움직이는 기능(애니메이션, 버튼 등)이 실제로 작동. |
| 외부 파일 | **불러옴** | HTML이 참조하는 같은 폴더의 디자인 파일·이미지가 함께 보임. |

즉, **실제 웹브라우저로 연 것과 똑같이** 보이게 하는 것이 목표입니다.

---

## 2. 왜 이런 구조가 필요한가

HTML 파일은 마크다운과 성격이 다릅니다.

- 마크다운: 글자 위주의 안전한 문서. 앱이 직접 변환해서 보여줘도 문제없음.
- HTML: **그 자체가 작은 웹페이지**. 디자인·움직이는 기능·외부 파일 연결을 다 담고 있음.

문제는, HTML 파일을 앱 화면에 그냥 그대로 펼치면 두 가지가 위험합니다.

1. **HTML의 디자인이 앱 화면을 망가뜨림** — HTML이 가진 색·글꼴·레이아웃이 앱 본체로 새어 나옴.
2. **HTML의 기능이 앱 권한을 훔칠 수 있음** — 악성 HTML 파일이라면 앱이 가진 "파일 읽기/쓰기/삭제" 권한을 몰래 사용할 수 있음.

그래서 HTML 파일은 **앱과 완전히 분리된 별도의 칸 안에** 보여줘야 합니다.

---

## 3. 어떻게 보여줄 것인가 (핵심 구조)

### 비유: "방탄 유리 부스 안에서 보여주기"

HTML 파일을 앱 화면에 직접 펼치지 않고, **격리된 작은 창(부스)** 안에 넣어서 보여줍니다. 이 부스는 웹 기술에서 `iframe`이라고 부르는 표준 장치입니다.

- 부스 안에서 HTML은 자유롭게 동작 (디자인·스크립트·이미지 다 작동) → 사용자가 원하는 "브라우저처럼".
- 부스 밖(앱 본체)으로는 아무것도 새어 나오지 못함 → 디자인 오염도, 권한 탈취도 차단.

이 부스에는 **"안전 잠금장치"**(`sandbox`)를 채워서, HTML이 할 수 있는 일을 딱 필요한 만큼(화면 그리기, 스크립트 실행)만 허용하고, 위험한 일(앱 본체 건드리기, 앱 권한 사용)은 막습니다.

### 동작 흐름

```
사용자가 사이드바에서 .html 파일 클릭
        │
        ▼
앱이 "이건 HTML이네" 하고 판단 (확장자로 구분)
        │
        ├─ 편집기는 "이 파일은 보기 전용입니다" 안내만 표시
        │
        ▼
프리뷰 칸에 격리된 부스(iframe)를 띄우고 그 안에서 HTML을 펼침
        │
        ▼
부스 안에서 HTML의 디자인·이미지·스크립트가 브라우저처럼 동작
```

### 한 가지 사전 작업이 필요함

지금 앱에는 "내 컴퓨터 안의 파일을 부스에 안전하게 전달하는 통로"가 꺼져 있습니다. (Tauri라는 앱 틀의 `asset 프로토콜`이라는 기능 — 현재 비활성화 상태)

이 통로를 켜야 HTML이 참조하는 디자인 파일·이미지가 함께 보입니다. 그리고 켤 때 \*\*"지금 열어둔 작업 폴더 안의 파일만 전달 가능"\*\*하도록 범위를 좁혀서, HTML이 컴퓨터의 엉뚱한 파일(비밀번호 파일 등)을 끌어오지 못하게 막습니다.

## 4. 위험 요소와 대비책

각 위험에 대해 "왜 위험한지"와 "어떻게 막는지"를 정리했습니다.

| 위험 | 쉬운 설명 | 심각도 | 대비책 |
| --- | --- | --- | --- |
| **앱 권한 탈취** | 악성 HTML이 앱의 파일 삭제·수정 권한을 몰래 사용 | 매우 높음 | 부스(iframe)가 앱과 완전히 분리된 공간이라 앱 권한에 손이 닿지 않음. **반드시 실제 공격 테스트로 확인.** |
| **다른 파일 훔쳐보기** | HTML이 컴퓨터의 다른 비밀 파일을 몰래 읽음 | 높음 | 안전 잠금장치 + "작업 폴더 안만 접근 허용" 범위 제한으로 차단. |
| **정보 빼돌리기** | HTML 스크립트가 읽은 내용을 외부 서버로 전송 | 높음 | 위 두 대비책으로 "읽을 수 있는 것 자체"가 작업 폴더로 한정됨. 다만 인터넷 연결 자체는 "스크립트 실행" 요구상 완전히는 못 막음 → **이 부분은 잔존 위험으로 사용자에게 안내 필요.** |
| **앱 화면 오염** | HTML의 디자인이 앱 본체 화면을 망가뜨림 | 낮음 | 부스 구조가 디자인을 완벽히 차단. |
| **앱 멈춤** | 너무 큰 HTML이나 무거운 스크립트가 앱을 느리게 함 | 중간 | 일정 크기(예: 5MB) 넘는 파일은 "미리보기 불가" 안내로 대체. |
| **경로 빠져나가기** | HTML이 `../../` 같은 방법으로 작업 폴더 밖 파일 접근 | 높음 | "작업 폴더 안만 허용" 범위 제한이 폴더 밖 접근을 막음. **반드시 테스트.** |
| **제품 방향성 충돌** | 이 앱은 "미니멀한 마크다운 전용 도구"로 소개되어 있고, 문서에 "HTML 미지원"이 명시됨 | 중간 (기술 아닌 제품 판단) | 이번 기능은 *독립 HTML 파일 보기*로 기존 "마크다운 안의 HTML"과는 다르지만, 앱 성격을 넓히는 것은 사실. **구현 전 제품 방향 확인 + 소개 문서 갱신 권장.** |

**가장 중요한 안전 원칙 한 가지**:앱 본체의 보안 설정은 **그대로 두고**, 새 기능에 필요한 최소한의 설정만 추가합니다. 위험은 "부스 안"으로만 가두고, 앱 전체로 번지지 않게 합니다.

## 5. 작업 항목 (무엇을 만들고 고치는가)

### 새로 만들 것

- **HTML 보기 화면** — 격리된 부스(iframe)를 띄우고 그 안에 HTML을 펼치는 부분.
- **파일 종류 분기 장치** — 클릭한 파일이 마크다운인지 HTML인지 판단해 알맞은 화면으로 보내는 부분.

### 고칠 것

- **사이드바 파일 목록** — 지금은 `.md` 파일만 보임 → `.html` 파일도 보이도록.
- **파일 열기 동작** — HTML 파일을 열면 편집기에 내용을 싣지 않고 "보기 전용" 안내만 표시.
- **편집기 화면** — HTML 파일일 때 "이 형식은 편집할 수 없습니다" 플레이스홀더 표시.
- **앱 보안/연결 설정** — 파일 전달 통로(asset 프로토콜)를 켜고, 부스를 허용하도록 설정. 범위는 작업 폴더로 제한.

### 핵심 파일 (개발자 참고용)

- `src-tauri/tauri.conf.json` — 보안·연결 설정
- `src/hooks/useFileSystem.ts` — 파일 열기 동작
- `src/components/preview/` — 미리보기 화면 (HtmlPreview, PreviewPanel 신규)
- `src/components/sidebar/FileExplorer.tsx` — 사이드바 파일 필터 (44행)
- `src/components/layout/AppLayout.tsx` — 화면 배치 연결

## 6. 검증 계획 (제대로 동작하는지 확인하는 법)

**테스트용 샘플 폴더** 하나를 만듭니다 — HTML 파일 + 그것이 참조하는 디자인 파일·이미지·스크립트 + 하위 폴더의 또 다른 HTML.

### 기능 확인

1. 폴더 열기 → 사이드바에 HTML 파일이 보이는가
2. HTML 클릭 → 디자인·이미지·움직이는 기능이 브라우저처럼 동작하는가
3. 마크다운 ↔ HTML 전환 → 화면이 올바르게 바뀌고, 편집기는 HTML일 때 보기 전용 안내가 뜨는가
4. 하위 폴더 HTML → 상위 폴더의 디자인 파일도 잘 불러오는가

### 보안 확인 (가장 중요)

5. **공격 시뮬레이션** — 일부러 "앱 권한을 훔치려는" 테스트 HTML을 만들어 열어봄 → 차단되어야 통과
6. **경로 탈출 시도** — 작업 폴더 밖 파일을 끌어오려는 HTML → 막혀야 통과
7. **다른 파일 읽기 시도** — 범위 밖 파일을 읽으려는 HTML → 막혀야 통과
8. **기존 기능 회귀** — 설정 변경 후에도 기존 마크다운·수식·다이어그램이 정상인가

### 성능 확인

 9. 매우 큰 HTML 파일 → "미리보기 불가" 안내가 뜨고 앱이 멈추지 않는가
10. 마크다운 ↔ HTML 빠르게 여러 번 전환 → 메모리가 새지 않는가

---

## 7. 권장 진행 순서

1. 앱 보안/연결 설정 변경 → 파일 전달 통로가 실제로 작동하는지 먼저 확인 (이게 안 되면 나머지 무의미)
2. HTML 보기 화면(부스) 제작 → **보안 테스트(5\~7번)부터 통과시키기**
3. 파일 종류 분기 + 화면 연결
4. 파일 열기 동작 + 편집기 보기 전용 처리
5. 사이드바 필터 확장
6. 작업 폴더로 접근 범위를 좁히는 추가 보안 작업 (권장)
7. 전체 검증 + 앱 소개 문서 갱신

## 참고: 결정 사항

- **제품 방향**: HTML 보기 기능을 독립 기능으로 추가하고, 구현 완료 후 product.md·README 소개 문서를 갱신한다. 마크다운 편집이라는 핵심은 그대로 유지. (확인 완료 — 진행)
- **잔존 위험 수용**: "스크립트 실행"을 허용하므로 악성 HTML이 인터넷으로 정보를 보내는 가능성은 완전히는 막을 수 없다. 단 읽을 수 있는 정보 자체는 작업 폴더로 제한되며, 신뢰할 수 있는 HTML 파일만 연다는 전제로 진행한다. (확인 완료 — 수용)
